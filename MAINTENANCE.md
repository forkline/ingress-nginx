# Maintenance Plan

This fork no longer follows the retired upstream project's release and CI model.
We maintain this repository as a GitHub-native project with `main` as the only long-
lived branch, simplified `vX.Y.Z` tags, GHCR for images and Helm charts, and
Renovate for dependency updates.

## Goals

- Keep the fork releasable from GitHub Actions only.
- Keep the maintenance surface small and explicit.
- Treat NGINX, Go modules, Kubernetes modules, and controller images as the main
  update lanes.
- Reuse the workflow style from `~/kaniop` where it lowers maintenance cost.

## Decisions

- Branch model: `main` only, plus release tags.
- Release tags: `vX.Y.Z`.
- Container registry: `ghcr.io/forkline/ingress-nginx`.
- Helm chart registry: `ghcr.io/forkline/helm-charts`.
- Dependency automation: Renovate.
- CI style baseline: `kaniop`.

## Target Workflow Set

The repository should converge on this workflow layout:

1. `pre-commit.yml`
2. `go.yml`
3. `e2e.yml`
4. `helm.yml`
5. `docker_images.yml`
6. `verify-generated.yml`
7. `security.yml`
8. `pages.yml`
9. `retry-workflow.yaml`

## Phase 1: Repository Hygiene

Status: partially implemented in this branch.

1. Add pre-commit hooks for whitespace, YAML, shell, Renovate config, and commit
   messages.
2. Add commitlint configuration with a small set of allowed scopes.
3. Add a dedicated `pre-commit.yml` workflow for `main` and pull requests.
4. Add a reusable retry workflow for flaky jobs.
5. Replace Dependabot with Renovate.

## Phase 2: Go CI Consolidation

Replace the current split between `ci.yaml` and `golangci-lint.yml` with a single
Go-focused workflow that:

1. Detects code changes and skips docs-only pull requests.
2. Runs unit tests with coverage tracking (see Issue #4).
3. Uploads coverage to Codecov for regression prevention.
4. Runs `golangci-lint`.
5. Verifies generated docs and manifests.
6. Builds controller artifacts.
7. Publishes a concise summary to the job summary.

Implementation notes:

- Keep current `make test`, `make verify-docs`, and build targets as the first
  migration step.
- Coverage tracking is critical for maintenance mode (see `.github/codecov.yml`).
- Coverage must never decrease from baseline to prevent regressions.
- Preserve path filtering from `.github/workflows/ci.yaml` while reducing the
  number of workflows a contributor has to understand.
- Add `concurrency` groups so superseded pull request runs are canceled.

## Phase 3: E2E Strategy

Build a dedicated `e2e.yml` instead of mixing e2e concerns into the main CI.

1. Pull requests:
   - Run one stable Kubernetes version.
   - Run one latest supported Kubernetes version.
2. Tags:
   - Run the full supported matrix.
3. NGINX base image changes:
   - Always run e2e.
4. Chart-only changes:
   - Run chart lint/tests plus one install smoke test.

The retry workflow should be wired into this job first, because this is the most
likely place to see non-deterministic failures.

## Phase 4: Helm Release Ownership

Replace the current chart release path that still assumes upstream layouts.

1. Keep lint and helm unit tests on pull requests.
2. Publish OCI charts to `ghcr.io/forkline/helm-charts/ingress-nginx` on tags.
3. Optionally add GPG signing once the release path is stable.
4. Add Artifact Hub metadata publication only after OCI publishing is working.

Implementation notes:

- Use `kaniop`'s `helm.yml` as the style template.
- Keep `ct lint` and helm unit tests from the existing repo.
- Remove assumptions tied to `kubernetes.github.io/ingress-nginx` where they are
  no longer correct for the fork.

## Phase 5: Image Release Ownership

Replace upstream-oriented image publishing with fork-owned GHCR publishing.

1. Publish controller images to `ghcr.io/forkline/ingress-nginx/controller`.
2. Publish the NGINX base image to `ghcr.io/forkline/ingress-nginx/nginx`.
3. Publish helper images only if they are still needed by the fork.
4. Use `vX.Y.Z` tags and `latest` for `main` only if you want rolling images.

Implementation notes:

- Adapt `kaniop`'s `docker_images.yml` rather than copying it literally.
- Keep multi-arch publishing only for images that are part of the supported
  release surface.
- Reduce image sprawl if some upstream helper images are no longer needed.

## Phase 6: Generated Artifact Drift Checks

Create `verify-generated.yml` with the same pattern `kaniop` uses for generated
example drift.

This workflow should fail if contributors forget to commit regenerated output for:

1. Helm README output from `helm-docs`.
2. Generated deploy manifests.
3. Annotation or API docs generated by existing scripts.

The implementation pattern is:

1. generate
2. `git diff --exit-code`
3. print a focused remediation message

## Phase 7: Security Baseline

Keep security checks that are actionable for a fork, and remove checks that exist
only because upstream used other infrastructure.

Recommended baseline:

1. dependency review on pull requests
2. Trivy scans for released images
3. Scorecards on `main`
4. `govulncheck` in Go CI

Recommended later:

1. cosign image signing
2. SBOM generation
3. provenance attestations

## Maintenance Lanes

The frozen-code assumption means recurring work should be handled in a few explicit
lanes.

### NGINX Lane

1. update `images/nginx/TAG`
2. update image build inputs in `images/nginx/rootfs`
3. update `NGINX_BASE`
4. run image build, smoke checks, and e2e

### Go and Kubernetes Lane

1. update `GOLANG_VERSION`
2. update `go.mod` and `go.sum`
3. run unit tests, lint, build, and e2e

### Security Lane

1. review Renovate pull requests
2. rebuild on base image CVEs
3. review Trivy and `govulncheck` output
